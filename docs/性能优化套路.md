# 性能优化套路（第一章实战总结）

> 2026-07-11 第一章优化实录。手机(iPhone,微信小游戏)从 **17 FPS → 稳 60**。
> 以后每个新篇章、每个新特效，按本文套路做，不要再走弯路。

## 一句话总纲

**动的东西用贴图精灵，静的东西画一次就缓存，热循环里不分配内存。**
Graphics 每帧 clear() 重画 = 每帧在 CPU 上重建几何 —— 只适合原型，上线前必须清干净。

## ⭐ 特效第一优先级：帧动画（序列帧贴图）

**做任何新特效，第一反应是「生成序列帧图」，而不是用 Graphics 画。**

### 工作流（跳劈冲击波实测流程）

1. **用 AI 生成序列帧**（Leonardo / GPT 图像均可），提示词模板：

   ```
   pixel art sprite sheet, 4 frames in a horizontal row, <特效内容>,
   frame-by-frame animation from <起始状态> to <结束状态>,
   pure black background, retro 16-bit game VFX, side view, no characters
   ```

   要点：
   - **横排 4 帧**（数量可调，4~6 帧足够）
   - **纯黑背景** —— 亮度转 alpha，不用抠图（抠图会丢细节，上次丢过武器）
   - 负面词：`character, weapon, text, watermark, gradient background`

2. **交给 Claude 处理**：检测帧簇 → 统一地面线切帧 → 黑底转 alpha（亮度=透明度）→ 256 色量化压缩 → 存 `assets/resources/fx-xxx.png`

3. **代码接入模板**（参考 `fx-slam-impact` 的实现，搜 `updateSlamFx`）：
   - `AssetHub.loadSF` 加载 → 按 `Rect(i*帧宽, 0, 帧宽, 帧高)` 切成 SpriteFrame 数组
   - 贴图设 `NEAREST` 采样（像素风）
   - 一个 Sprite 节点 + 计时器选帧，触发时 `slamFxT = DUR`
   - 贴地特效用**非等比缩放**（横大纵小，如 2.5×1.2）做透视贴地感

### 已用此法完成的特效（可当模板抄）

| 特效 | 贴图 | 实现 |
|---|---|---|
| 跳劈落地冲击波 | fx-slam-impact（AI 生成 4 帧） | `updateSlamFx` 计时选帧 |
| 刀气/剑气波 | fx-crescent（代码烘焙新月） | 单精灵转角度+淡出 |
| 血滴/尘土 | fx-dot（14px 圆点） | 精灵池 `dotDraw`，同贴图合批=1 draw call |
| 雨/雪/落叶/浮尘 | fx-rain/snow/leaf-fall/mote 平铺图 | 无缝平铺层滚动，每帧 1 次 setPosition |

## 五板斧（按收益排序，附实测数字）

### 1. 脏标记/缓存 —— 收益最大
静态或低频变化的层，**内容没变就不重画**。
- 背景天空+地面：屏幕固定坐标 + 只按时段重画 → Game Logic **20ms → 5ms**（决定性一刀）
- HUD 血条：血量量化成 key，变了才重画（平时零开销）
- 云：位置吸附 12px 网格，飘过一格才重画（~1次/秒）
- 地面残留(血渍/脚印/雪面)：独立层 8Hz + 卷屏 6px 步进
- 技能冷却环：48 级量化
- **陷阱**：key 里别放 camX 这种战斗中一直变的量（第一版缓存战斗时失效，白做）

### 2. 粒子贴图化 —— 平铺层滚动
雨 70 滴×3 块 = 每帧 150 个 rect → **2 张平铺贴图斜向滚动 = 每帧 2 次坐标赋值**。
- 贴图周期要大（384×512），粒子撒够数量，否则平铺重复会排成整齐的行（踩过坑）
- Mask 裁切到地面线，粒子不穿地
- 落地溅花用定时器随机补，代替逐粒落地判定

### 3. 特效精灵化 —— 见上文帧动画

### 4. 降频更新
纯氛围动画（叶摆/草摆/小动物/水花）**隔帧更新，dt×2 补偿** → 这组 CPU 砍半，视觉无感。
快速移动的东西（雨层滚动）保持满帧，否则有跳感。

### 5. 零分配热循环
- `new Color(...)` → 复用 `this._scratchC.set(...)`（setter 都是值拷贝，安全）
- `[...arr].sort(闭包)` → 复用数组 + 静态比较器
- 字符串 key 拼接 → 数字哈希
- 按关不变的计算（charTint/gradeColor）→ 按 zone 缓存

## 微信小游戏发布要点

- **iOS 高性能模式是最大的一颗银弹**：普通模式 JS 无 JIT，比桌面慢 20~50 倍；
  开启后 20 FPS → 60 FPS。Cocos 构建面板没有此选项，**每次构建后双击项目根目录的
  `构建后补丁.command`**（自动加 iOSHighPerformance + 关编译拖慢项 + 校验 AppID）
- 真机调试比预览慢很多，性能以「预览」和「体验版」为准
- 微信容器帧率上限 60，别追 120
- 兜底方案：稳不住 60 就 `game.frameRate = 30`（稳 30 手感好于波动的 20~40）

## 性能验证方法

- `BattleScene.SHOW_FPS = true`：屏上 FPS+敌人数（发布前改 false）
- `profiler.showStats()`：看 **Game Logic(ms)** = JS/几何重建瓶颈；**Renderer(ms)** = GPU/渲染瓶颈；Draw call 健康值 < 100
- **先测数字再动手**，别猜。这次前三刀（降分辨率/砍overdraw）全打空，就是因为没先看 profiler
